## Context

현재 `/api/search/files`는 raw 배열만 반환하고, 프론트는 헤더 검색과 검색 페이지 모두 이 배열만 해석합니다. 검색 결과는 이름 기준 일치도 순으로 정렬되지만, 사용자는 결과 수, 부모 경로, 잘림 여부를 알 수 없고, 백엔드는 요청마다 접근 가능한 스페이스를 순회하며 `WalkDir`를 수행합니다.

이번 change는 검색 전체 구조를 갈아엎지 않고, 기존 엔드포인트와 정렬 규칙을 유지하면서 응답 계약과 화면 문맥을 보강하는 1차 슬라이스다. 외부 인덱스나 cursor 기반 페이지네이션은 범위에서 제외하고, 현재 구조에서 안전하게 구현 가능한 `hasMore`와 단계적 더 보기 흐름에 집중한다.

## Goals / Non-Goals

**Goals:**
- `/api/search/files` 응답을 메타데이터 포함 객체로 확장한다.
- 헤더 검색 드롭다운과 검색 페이지가 스페이스/부모 경로를 보여주도록 한다.
- 검색 페이지가 반환된 결과 수, 정렬 기준, 잘림 여부, 더 보기 액션을 제공하도록 한다.
- 현재 권한 체크와 정렬 규칙을 유지한 채 테스트를 보강한다.

**Non-Goals:**
- 전체 텍스트 검색 도입
- 외부 검색 엔진 또는 인덱스 도입
- cursor/token 기반 서버 페이지네이션
- 검색 랭킹 알고리즘 자체 변경

## Decisions

### 1. 응답 계약은 `items + limit + hasMore` 객체로 확장한다.
- 결정: 검색 API는 `items`, `limit`, `hasMore`를 반환한다.
- 이유: 프론트가 잘림 여부와 현재 상한을 직접 해석할 수 있어야 한다.
- 대안:
  - `total`까지 계산: 현재 구조에서는 전체 스캔 비용이 커지고 성능 목표와 충돌한다.
  - raw 배열 유지: UI가 잘림 여부를 알 수 없어 추가 탐색 경로를 제공할 수 없다.

### 2. `hasMore`는 limit 초과 여부만 판단하고, 절대 총 개수는 제공하지 않는다.
- 결정: 백엔드는 `limit + 1`개째 일치 항목을 감지하면 `hasMore=true`로 응답하고 결과는 `limit`개까지만 반환한다.
- 이유: 전체 개수를 세지 않고도 사용자가 결과가 더 있다는 사실을 알 수 있다.
- 대안:
  - 모든 결과 개수 계산: 응답 지연이 커진다.
  - `hasMore` 없이 limit만 노출: 잘림인지 정확히 알 수 없다.

### 3. 검색 페이지의 추가 탐색은 `더 보기` 기반 단계적 limit 증가로 구현한다.
- 결정: 검색 페이지는 내부 limit 상태를 80개 단위로 늘려 재조회한다.
- 이유: 현재 엔드포인트와 정렬 규칙을 유지한 채 추가 탐색 경로를 만들 수 있다.
- 대안:
  - 서버 offset/cursor 추가: 현재 정렬/수집 구조에서 복잡도가 커진다.
  - 추가 탐색 경로 없음: 이슈 AC를 충분히 만족하지 못한다.

### 4. 결과 문맥은 `spaceName + parentPath`를 공통 최소 단위로 노출한다.
- 결정: 헤더 드롭다운과 검색 페이지 모두 `spaceName`과 `parentPath`를 표시한다. 검색 페이지는 기존 크기/수정일 메타에 경로를 추가한다.
- 이유: 동일한 파일명 간 구분에 가장 직접적이고 현재 응답 필드로 즉시 구현 가능하다.
- 대안:
  - 전체 경로만 표시: 길이가 길어져 가독성이 떨어진다.
  - 스페이스만 표시: 동일 스페이스 내 중복 파일명을 구분하기 어렵다.

### 5. 정렬 기준은 UI 설명으로만 노출하고 알고리즘은 유지한다.
- 결정: 검색 페이지에 "이름 일치도 순, 동일 시 스페이스/경로 순" 안내를 추가한다.
- 이유: 정렬 기대치를 사용자에게 설명하면서 랭킹 변경 리스크를 피한다.
- 대안:
  - 알고리즘 변경: 범위가 커지고 회귀 가능성이 높다.

## Risks / Trade-offs

- [검색 페이지 더 보기 재조회 비용] → limit을 늘릴 때 처음부터 다시 검색하므로 큰 쿼리에서 비용이 증가한다. 1차 슬라이스에서는 구현 복잡도보다 단순성을 우선한다.
- [응답 스키마 변경 영향] → 프론트 API 래퍼와 테스트를 함께 업데이트해 계약 파손을 조기에 감지한다.
- [헤더 드롭다운 정보 과밀] → 부모 경로는 한 줄 요약 또는 보조 메타로 제한해 시각 밀도를 관리한다.
