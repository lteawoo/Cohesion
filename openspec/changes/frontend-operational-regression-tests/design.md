## Context

프론트엔드 테스트는 현재 파일 탐색, 사이드바, 감사 로그, 스페이스 설정 같은 일부 조각만 커버한다. 반면 로그인, 설정 페이지 권한 분기, 메인 레이아웃 헤더 검색, 서버 설정은 실제 운영 동선임에도 회귀 테스트가 비어 있다. 이 영역들은 라우팅, 권한, debounce, 비동기 API, 메시지/모달 등 상호작용이 많아 테스트를 추가할 때도 flaky를 피하는 설계가 필요하다.

## Goals / Non-Goals

**Goals:**
- 로그인, 설정 페이지, 헤더 검색, 서버 설정의 핵심 분기를 자동 테스트로 고정한다.
- 기존 Vitest + RTL 패턴과 file-local `vi.mock` 구조를 재사용한다.
- 타이머와 비동기 요청이 있는 경로는 최소 핵심 분기만 고정해 반복 실행 가능한 테스트를 만든다.

**Non-Goals:**
- E2E 프레임워크 도입
- 공용 테스트 프레임워크 전면 리팩터링
- 제품 동작 변경이나 권한 정책 변경
- 서버 설정의 모든 재시작 분기 완전 시뮬레이션

## Decisions

### 1. 테스트는 화면별 독립 파일로 추가한다
- 선택: `Login`, `SettingsPage`, `MainLayout`, `ServerSettings` 각각의 테스트 파일을 둔다.
- 이유: 현재 코드베이스의 테스트가 file-local mock과 작은 render helper를 쓰고 있고, 이 스타일이 가장 낮은 결합도로 유지된다.
- 대안: 공용 `renderWithProviders` 도입
- 배제 이유: 이번 이슈 범위는 회귀 안전망 확보이지 테스트 인프라 리팩터링이 아니다.

### 2. 헤더 검색은 `MainLayout` 렌더를 통해 검증한다
- 선택: 내부 `HeaderSearch`를 export하지 않고 `MainLayout` 수준에서 상호작용을 검증한다.
- 이유: 테스트를 위해 프로덕션 심볼을 노출하지 않고도 현재 계약을 검증할 수 있다.
- 대안: `HeaderSearch`를 분리/export
- 배제 이유: 테스트 편의를 위한 구조 변경이 불필요한 리팩터링을 유발한다.

### 3. 서버 설정은 핵심 분기만 고정한다
- 선택: 로드 실패, validation 차단, 저장 성공/실패, 재시작 confirm 및 개발 모드 재연결 성공/실패 정도만 테스트한다.
- 이유: `window.location`, `waitForReconnect`, 재시작 지연까지 모두 깊게 묶으면 flaky 가능성이 커진다.
- 대안: 프로덕션/개발 분기와 리다이렉트까지 전부 커버
- 배제 이유: 비용 대비 가치가 낮고 유지보수성이 떨어진다.

### 4. 권한 기반 설정 메뉴는 노출 계약 중심으로 검증한다
- 선택: 권한별 메뉴 노출/비노출과 섹션 전환을 검증한다.
- 이유: 접근 불가 섹션 fallback은 UI에서 직접 진입하지 않는 내부 방어 로직이라 우선순위가 낮다.
- 대안: internal state를 조작해 fallback까지 직접 테스트
- 배제 이유: 구현 결합이 커지고 실사용 회귀 감지 가치가 낮다.

## Risks / Trade-offs

- [MainLayout 모킹 범위가 넓음] → 하위 컴포넌트를 단순 stub로 대체하고 검색 동작에만 집중한다.
- [debounce/타이머 기반 flaky] → `vi.useFakeTimers()`를 헤더 검색 테스트에만 한정 사용한다.
- [서버 설정 재시작 테스트의 환경 의존성] → `api/config`를 완전히 mock하고 실제 네트워크/리로드는 호출 여부만 검증한다.
- [권한 정책이 바뀌면 테스트 다수 수정 필요] → 현재 정책을 spec로 고정하고, 정책 변경은 별도 이슈로 분리한다.
