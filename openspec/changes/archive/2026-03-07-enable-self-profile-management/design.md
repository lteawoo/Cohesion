## Context

현재 프로필 화면은 읽기 전용이고, 계정 수정은 `PATCH /api/accounts/{id}`를 통한 관리자 전용 경로만 존재한다. 반면 권한 모델과 시드 데이터에는 `profile.read`, `profile.write`가 이미 정의돼 있어 자기수정 권한의 개념은 준비돼 있지만 실제 라우트와 UI가 비어 있다. 이번 변경은 프론트 `ProfileSettings`, 인증 컨텍스트, 백엔드 auth/account 경계를 함께 건드리므로 구현 전에 자기수정 경로와 세션 동기화 기준을 먼저 고정할 필요가 있다.

## Goals / Non-Goals

**Goals:**
- 로그인 사용자가 `Settings > 내 프로필`에서 자신의 닉네임을 수정할 수 있게 한다.
- 로그인 사용자가 현재 비밀번호를 확인한 뒤 새 비밀번호를 변경할 수 있게 한다.
- 자기수정 성공 후 `/api/auth/me` 기반 세션 상태를 최신 값으로 동기화한다.
- 자기수정 API를 `profile.write` 권한과 연결해 관리자용 `account.write` 경로와 분리한다.
- 프론트/백엔드 회귀 테스트로 주요 성공/실패 경로를 고정한다.

**Non-Goals:**
- 초기 admin setup 흐름 변경
- 관리자용 계정 관리 화면 정책 변경
- 사용자명 변경, 역할 변경, MFA/OAuth 같은 인증 체계 확장
- 비밀번호 정책 전면 개편

## Decisions

### 1. 자기수정 API는 `PATCH /api/auth/me`로 분리한다
- 이유: `/api/accounts/{id}`는 관리자용 계정 관리 경로이며 `account.write`에 묶여 있다. 자기수정은 리소스 대상이 “현재 로그인 사용자”이므로 `auth` 하위 경로가 더 자연스럽고 권한도 `profile.write`로 분리하기 쉽다.
- 대안: 기존 `/api/accounts/{id}`를 자기 자신에게도 허용
- 대안 기각 사유: 일반 사용자에게 `account.write` 경계를 열거나, 예외 규칙을 핸들러 내부에 추가해야 해 권한 모델이 흐려진다.

### 2. 비밀번호 변경 시에만 `currentPassword`를 필수로 요구한다
- 이유: 닉네임만 수정할 때까지 현재 비밀번호를 매번 요구하면 UX 비용이 과하다. 반면 비밀번호 변경은 계정 보안 액션이므로 현재 비밀번호 검증이 필요하다.
- 대안: 모든 프로필 저장에 현재 비밀번호 요구
- 대안 기각 사유: 닉네임 변경 같은 저위험 액션까지 불필요하게 복잡해진다.

### 3. 자기수정은 닉네임/비밀번호만 허용하고 역할/아이디 변경은 금지한다
- 이유: 현재 세션과 권한 모델에 영향을 주는 필드는 관리자 경로로 한정하는 편이 안전하다.
- 대안: 역할 변경까지 자기수정 허용
- 대안 기각 사유: 권한 상승/하향과 마지막 관리자 보호 규칙까지 자기수정 경로가 끌어오게 된다.

### 4. 저장 성공 후 프론트는 `refreshSession()`으로 전역 사용자 상태를 갱신한다
- 이유: auth 미들웨어가 매 요청마다 현재 DB 사용자 정보를 claims에 재주입하므로, 프로필 저장 후 `me()`를 다시 읽으면 닉네임이 즉시 최신화된다.
- 대안: 저장 응답으로 받은 사용자로 컨텍스트를 직접 덮어쓰기
- 대안 기각 사유: 세션 응답 계약과 중복 상태를 만들고, 이후 권한/역할 반영 정책과 어긋날 수 있다.

### 5. 비밀번호 길이 정책은 자기수정 경로에서 8자 이상으로 올리고 초기 setup은 유지한다
- 이유: 초기 setup 프론트는 이미 8자 이상을 요구한다. 자기 비밀번호 변경도 같은 기준으로 맞추면 사용자 관점 정책이 일관된다.
- 대안: 기존 계정 수정 서비스처럼 6자 이상 유지
- 대안 기각 사유: 로그인/초기 setup과 자기수정 사이 정책이 계속 어긋난다.

## Risks / Trade-offs

- [관리자 계정 수정과 자기수정 비밀번호 정책 불일치] → 이번 변경은 자기수정 경로만 8자 기준으로 올리고, 관리자용 타인 계정 수정 정책은 별도 이슈에서 정리한다.
- [권한 매핑 누락으로 자기수정 경로가 403 또는 무보호 상태가 되는 위험] → `requiredPermissionForRequest`와 denied audit 매핑 테스트를 함께 추가한다.
- [프로필 저장 후 화면 상태 불일치] → `refreshSession()` 호출과 프로필 화면 테스트로 저장 직후 렌더 갱신을 고정한다.
- [현재 비밀번호 검증 로직 중복] → 새 암호 검증은 auth 서비스에서 현재 사용자 조회 + account 서비스 update 호출을 조합하는 최소 경로로 추가한다.
