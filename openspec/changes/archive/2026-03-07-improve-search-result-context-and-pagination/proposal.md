## Why

현재 파일 검색은 헤더 제안과 검색 페이지가 동일한 배열 응답 API를 공유하고 있어 결과 수, 잘림 여부, 경로 문맥을 충분히 전달하지 못합니다. 검색 대상이 늘어나기 전에 응답 계약과 UI 문맥을 먼저 정리해야 사용자가 결과를 비교하고 추가 탐색 여부를 판단할 수 있습니다.

## What Changes

- 검색 API 응답을 raw 배열에서 메타데이터를 포함한 객체로 확장한다.
- 헤더 검색과 검색 페이지가 같은 응답 계약을 사용하되, 각 화면에 맞는 limit와 잘림 안내를 명확히 표시한다.
- 검색 결과에 스페이스와 부모 경로를 노출해 동일한 파일명 사이에서 문맥을 구분할 수 있게 한다.
- 검색 페이지에 결과 수, 잘림 안내, 추가 탐색 경로를 제공한다.
- 검색 핵심 경로의 프론트엔드/백엔드 테스트를 보강한다.

## Capabilities

### New Capabilities
- `file-search-result-exploration`: 파일 검색 응답 계약, 결과 문맥 표시, 잘림/추가 탐색 안내를 정의한다.

### Modified Capabilities
- None.

## Impact

- Backend: `/api/search/files` 응답 스키마와 제한 처리, 검색 테스트
- Frontend: 헤더 검색 드롭다운, 검색 페이지 상태 관리, 검색 결과 메타 렌더링
- Tests: 검색 API 계약, 헤더 검색, 검색 페이지 결과/상태 표시 회귀 테스트
